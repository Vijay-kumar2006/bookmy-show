# Design Decisions

## Decision: Concurrency Strategy (Redis SETNX vs FOR UPDATE)
 
**Context:** At 500,000 concurrent users attempting to book seats, the system must guarantee absolutely zero double-bookings.
 
**Options considered:**
1. *PostgreSQL SELECT FOR UPDATE*: Pessimistic row locking directly in the DB. Not chosen because connections are blocked while waiting for transactions, limiting scale.
2. *Redis SETNX Distributed Lock* - chosen. 
 
**Why chosen:** PostgreSQL connection pools (PgBouncer max ~500) will exhaust at roughly 2,840 RPS due to held transactions during payment or locking phases. A 3-node Redis cluster can effortlessly handle 100k+ lock acquisitions per second, protecting the expensive DB connections. We combine this with Optimistic Locking in Postgres as a final safety net.
 
**Tradeoffs accepted:** We introduce an extra infrastructural dependency (Redis). If Redis loses a write during failover, we could have a ghost hold or rely on the DB fallback, increasing complexity slightly.
 
**Revision trigger:** Moving to a massively scalable distributed SQL system (like Google Spanner or CockroachDB) that handles locking without a monolithic connection bottleneck.

## Decision: Cache Invalidation Approach
 
**Context:** Caching seat availability to prevent overwhelming the primary database with identical aggregate read queries (e.g., "how many VIP seats left?").
 
**Options considered:**
1. *Event-Driven Invalidation (Cache-Aside)*: Deleting the cache key every time a seat is held or booked. Not chosen for availability counts because at 25,000 RPS, this creates excessive cache churn and thundering herds.
2. *TTL-only Expiry (30s)* - chosen for aggregate counts.
 
**Why chosen:** A 30-second TTL drastically reduces database read load while providing "eventually consistent" accuracy that users accept for high-demand ticket sales. For static data (event details), we use Cache-Aside.
 
**Tradeoffs accepted:** The user interface might show seats as "available" that were held 15 seconds ago. Users might click and fail to acquire the lock.
 
**Revision trigger:** If business requirements mandate strict 100% real-time UI accuracy at all times, we would switch to a WebSocket push model driven by database CDC (Change Data Capture) via Debezium.

## Decision: UUID vs SERIAL for Booking IDs
 
**Context:** Every order requires a unique identifier for tracking, payment processing, and user reference.
 
**Options considered:**
1. *SERIAL (Auto-incrementing integer)*: Handled entirely by the DB. Simple, but exposes business intelligence (competitors can see how many tickets we sold) and complicates distributed creation.
2. *UUID (Universally Unique Identifier)* - chosen.
 
**Why chosen:** UUIDs can be securely generated by the Node.js API *before* inserting into the database. This is critical for our async payment architecture because we need to generate an `idempotencyKey` and `bookingId` to pass to SQS before confirming the DB write, allowing safe retries if the worker crashes.
 
**Tradeoffs accepted:** UUIDs are 128-bit, which are larger than 32-bit integers, slightly increasing DB storage space and index sizes, leading to minor performance degradation on B-Tree index traversal.
 
**Revision trigger:** If storage or index performance becomes a severe issue and we don't need distributed generation, or if we adopt a Snowflake-like distributed integer generation service (like Twitter Snowflake).

## Decision: SQS Visibility Timeout Value
 
**Context:** We use SQS to asynchronously process payments. When a worker pulls a message, SQS hides it from other workers (Visibility Timeout).
 
**Options considered:**
1. *10 seconds*: Shorter timeout means faster retries if a worker crashes. Not chosen because some payment gateways (Stripe/Razorpay) can take up to 15 seconds during peak loads.
2. *30 seconds* - chosen.
 
**Why chosen:** 30 seconds gives the payment worker ample time to handle maximum latency from a slow third-party payment gateway, update the database, and delete the message. It avoids the catastrophic scenario where a slow payment causes the message to reappear, getting processed by a second worker (duplicate charge).
 
**Tradeoffs accepted:** If a worker crashes instantly at second 1, the message sits invisible for 29 seconds before another worker can retry it, slightly delaying the user's confirmation.
 
**Revision trigger:** If we switch to a faster payment provider with guaranteed p99 sub-second latency, we would lower the timeout to ~5 seconds for faster crash recovery.
